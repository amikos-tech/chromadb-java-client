**INFRA-1: Remove branches filter**

Change the `on:` trigger block from:
```yaml
on:
  release:
    types: [created]
    branches: [ "main" ]
```
to:
```yaml
on:
  release:
    types: [created]
```

This removes the misleading `branches: ["main"]` filter. GitHub Actions ignores `branches:` on `release:` events (it only applies to `push:` and `pull_request:` events). Removing it makes the YAML accurate and eliminates false confidence about branch restriction.

**INFRA-2: Add release-check step**

Insert a new step AFTER the "Version bump" step and BEFORE the "Publish package" step:
```yaml
      - name: Run release-check
        run: make release-check
```

IMPORTANT: The step goes AFTER Version bump, not before it. The research (Pattern 4 / Pitfall 1 correction in 06-RESEARCH.md) explains why: `release-check` validates that CHANGELOG.md and README.md reference the release version (e.g., `0.2.0`). The Version bump step changes pom.xml from SNAPSHOT to the actual release version via `versions:set -DnewVersion=${{ github.ref_name }}`. So `release-check` must run against the bumped (non-SNAPSHOT) version to validate correctly. If it ran before Version bump, it would fail with "Version contains -SNAPSHOT".

The final step order in release.yml should be:
1. `actions/checkout@v4`
2. `Set up JDK 8`
3. `Install gpg secret key`
4. `Run unit tests`
5. `Run integration tests`
6. `Version bump`
7. `Run release-check`    (NEW)
8. `Publish package`

Change the version from `1.0.0-M2` to `1.0.0-M2.1`:
```xml

    <groupId>org.nd4j</groupId>
    <artifactId>nd4j-native-platform</artifactId>

```

This is a safe patch upgrade (M2 to M2.1). Per the research:
- 1.0.0-M2.1 is the latest published version on Maven Central
- The transitive dependency graph is nearly identical to M2
- The ARM64 `libjnind4jcpu.dylib` UnsatisfiedLinkError is NOT fixed in M2.1 (it's an inherent nd4j 1.x limitation on Apple Silicon)
- The existing `Assume.assumeNoException` guard in `TestDefaultEmbeddings.checkNd4jAvailable()` (commit c3224b6) handles ARM64 gracefully by skipping
- CI runs on `ubuntu-latest` (x86_64), so no CI impact

After changing the version, run `mvn dependency:tree -q 2>&1 | grep nd4j` to verify the resolved version and confirm no dependency conflicts.

Also run `mvn test-compile -q` to verify the project still compiles cleanly with the new version.
